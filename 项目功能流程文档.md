# 卧龙项目功能流程文档

## 1. 文档范围

本文档基于当前仓库实际代码整理，覆盖以下三部分：

- `wolong-frontend`：Vue 3 前端工作流界面
- `backend`：FastAPI + SQLite + Huey 后端
- `external/MediaCrawler`：外部集成的多平台公开信息采集器

本文档不是单纯复述 PRD，而是说明“当前项目实际上怎样工作”。

## 2. 项目定位

卧龙项目试图把一个决策问题拆成 6 个连续阶段：

1. 创建案例
2. 生成搜索策略并配置信息源
3. 采集公开内容并提炼事实
4. 审计事实并构建因果沙盘
5. 注入个人约束并推演世界线
6. 生成破局白皮书

从产品意图上看，它是一套“公开信息采集 + 大模型提炼 + 决策推演”的认知增强系统。

## 3. 总体架构

### 3.1 前端

- 使用 Vue 3 + TypeScript + Vite
- 使用 Pinia 管理少量全局状态
- 使用 Vue Router 组织多步骤工作流
- 使用 Element Plus 提供表单、表格、抽屉、对话框
- 使用 ECharts 渲染因果图和世界线树图

### 3.2 后端

- 使用 FastAPI 提供 HTTP API
- 使用 SQLAlchemy + SQLite 持久化案例、事实、证据、图谱和白皮书
- 使用 Huey 组织采集异步任务
- 使用 OpenAI 兼容接口调用外部大模型

### 3.3 外部爬虫

- 通过 `external/MediaCrawler` 运行真实平台爬虫
- 当前项目以命令行方式调用该仓库的 `main.py`
- 采集结果落地为 `backend/data/crawler/<case_id>/<platform>/jsonl/*.jsonl`

## 4. 目录分工

### 4.1 `backend`

- `app/main.py`：FastAPI 启动入口
- `app/api/routes.py`：所有业务 API
- `app/models/entities.py`：数据库实体
- `app/services/llm.py`：LLM 调用与 JSON 解析
- `app/services/crawler.py`：MediaCrawler 启动和 JSONL 解析
- `app/services/simulation.py`：沙盘/世界线干预推演
- `app/services/seed_data.py`：默认演示数据兜底
- `app/workers/tasks.py`：异步采集主任务

### 4.2 `wolong-frontend`

- `src/router/index.ts`：工作流路由
- `src/api/case.ts`：前端 API 封装
- `src/store/caseStore.ts`：案例与流程阶段状态
- `src/store/configStore.ts`：模型服务商与阶段绑定配置
- `src/views/*`：每个步骤对应一个页面
- `src/components/*`：每个页面使用的交互组件

### 4.3 `external/MediaCrawler`

- `main.py`：爬虫总入口，按平台分发
- `media_platform/*`：各平台采集实现
- `store/*`：各平台内容保存逻辑
- `config/*`：平台级采集配置

## 5. 端到端功能流程

## 5.1 案例创建

用户从首页输入一个“决策情景描述”，点击“一键生成决策案例”。

实际流程：

1. 前端 `CaseCreateView` 调用 `POST /cases`
2. 后端生成 `case_id`
3. 后端写入 `cases` 表
4. 后端立即调用 `ensure_case_seed_data`
5. 默认事实、证据、沙盘、世界线、白皮书会被一并写入数据库
6. 前端跳转到案例工作流页

结论：当前系统在“创建案例”时就已经预置了一套演示数据，这意味着后续即使真实采集未成功，页面也可能仍然有内容可看。

## 5.2 搜索策略生成

用户进入“搜商策略与信息源配置”页面后，系统自动触发关键词生成。

实际流程：

1. 前端 `StrategyConfigView` 挂载时调用 `POST /cases/{case_id}/search-strategy`
2. 请求头会自动携带本地保存的 `X-Wolong-Config`
3. 后端根据阶段绑定取出 `collection` 模型服务商
4. 后端用 `SEARCH_STRATEGY_PROMPT` 调用大模型
5. 生成的关键词写入 `search_strategies` 表
6. 页面允许用户手动新增、删除关键词
7. 页面允许用户选择平台、抓取上限、是否抓评论

说明：

- 如果没有配置模型服务商或 API Key，这一步会直接失败
- 前端允许改关键词，但当前后端真正使用的仍是数据库中的 `SearchStrategy.keywords`

## 5.3 启动采集任务

用户点击“立刻开始采集”后，进入采集进度页。

实际流程：

1. 前端调用 `POST /cases/{case_id}/start-collection`
2. 后端新建 `collection_tasks` 记录，并把案例状态改为 `COLLECTING`
3. 后端把前端传来的平台配置与请求头中的全局配置合并
4. 后端调用 Huey 任务 `run_collection(case_id, merged_config)`
5. 前端跳转到 `CollectionProgressView`
6. 页面每 800ms 轮询 `GET /cases/{case_id}/collection-status`

说明：

- README 要求单独启动 `huey_consumer`
- 若未启动 Huey worker，采集任务可能不会真正执行

## 5.4 真实采集与证据入库

`run_collection` 是整个数据管道的核心。

实际流程：

1. 读取关键词
2. 读取启用的平台列表
3. 对每个平台、每个关键词调用 `run_media_crawler`
4. `run_media_crawler` 用子进程执行 `external/MediaCrawler/main.py`
5. MediaCrawler 把结果写成 JSONL
6. 后端扫描 JSONL 并解析为原始条目
7. 后端把条目转换为 `Evidence`
8. 后端对内容做三类轻量过滤：
   - 文本 hash 去重
   - 含“广告/推广”的内容判为软广
   - 含“垃圾/吐槽”的内容判为情绪发泄
9. 过滤后的内容写入 `evidences` 表

说明：

- 过滤逻辑目前是关键字规则，不是语义级判断
- 证据文本主要由标题、正文、描述字段拼接得到

## 5.5 事实提炼与审计

设计上，证据入库后应调用大模型抽取事实卡片并附带审计提示。

当前代码中的目标流程：

1. 读取本案例下全部 `Evidence`
2. 分批调用 `generate_facts`
3. 写入 `facts` 表
4. 写入 `fact_audit_alerts` 表
5. 前端 `FactAuditView` 拉取事实卡片进行人工筛选和反馈

前端交互流程：

1. `FactsPoolView` 展示信息池摘要卡片
2. 用户进入 `FactAuditView`
3. 每张卡片可标记“有价值”或“低价值”
4. 点击“审计证据溯源”会打开抽屉
5. 抽屉调用 `GET /cases/{case_id}/evidences?ids=...`
6. 后端返回底层证据片段，前端用 `v-html` 呈现高亮内容

说明：

- `feedback` 数据最终写入 `fact_feedbacks`
- 当前“认知审计”更多是前端交互层完成，后端没有独立审计工作流状态机

## 5.6 因果沙盘

用户从事实审计页进入因果沙盘页，查看节点和边，并执行 What-if 推演。

实际流程：

1. 前端 `CausalSandboxView` 调用 `GET /cases/{case_id}/sandbox`
2. 后端优先读 `sandbox_graphs`
3. 如果图不存在或为空，则尝试调用大模型生成
4. 若仍失败，则回退到 `seed_data` 中的默认图谱
5. 前端用 ECharts Graph 显示节点和边
6. 用户可点击橙色“行动变量”节点进行滑杆干预
7. 用户也可以直接输入自然语言假设
8. 前端调用 `POST /cases/{case_id}/sandbox/intervene`
9. 后端执行 `simulate_sandbox`
10. 结果写入 `sandbox_interventions`
11. 前端将受影响节点的数值更新到图上，并展示逻辑解释和风险提示

说明：

- 当前推演既支持 LLM，也支持规则兜底
- 规则兜底是写死的 demo 逻辑，并不是真正通用的因果传播

## 5.7 世界线演化

用户进入世界线页面后，系统尝试生成多条时间线分支。

设计目标流程：

1. 用户先注入个人约束
2. 系统在事实 + 因果图 + 个人语境基础上推演未来分支
3. 用户可继续输入新决策文本干预世界线

当前实际流程：

1. 页面挂载时先调用 `GET /cases/{case_id}/worldline`
2. 如果数据库里已有 `worldlines` 数据，则直接展示
3. 页面理论上支持先弹出个人语境对话框
4. 对话框提交后会调用 `PUT /cases/{case_id}/context`
5. 随后再次调用 `GET /cases/{case_id}/worldline`
6. 用户输入自然语言干预后，前端调用 `POST /cases/{case_id}/worldline/intervene`
7. 后端执行 `simulate_worldline`
8. 结果写入 `worldline_interventions`
9. 前端将新的路径替换到时间树上

说明：

- 当前世界线树是把每条路径渲染成线性链表，再挂到根节点下
- “撤销”只是前端恢复上一份快照

## 5.8 白皮书生成

用户点击“生成破局白皮书”后进入报告页。

实际流程：

1. 世界线页面按钮本身不调用后端生成接口，只做 1.2 秒前端延时后跳转
2. 白皮书页挂载时调用 `GET /cases/{case_id}/whitepaper`
3. 后端优先读取 `whitepapers` 表
4. 若缺失，则尝试用事实 + 世界线调用大模型生成
5. 若仍失败，则回退为默认演示白皮书
6. 前端展示：
   - 核心矛盾
   - 临界预警
   - 未知数
   - MVP 行动清单
7. 用户可以对行动建议做负反馈
8. 白皮书支持使用 `html2canvas + jsPDF` 导出 PDF

说明：

- 反馈最终写入 `case_feedbacks`
- PDF 导出是整页截图式导出，不是结构化排版导出

## 5.9 历史案例与模型配置

### 历史案例

1. `HistoryView` 调用 `GET /cases/history`
2. 后端按创建时间倒序返回案例列表
3. 页面支持点击恢复到对应流程节点
4. 页面里的“删除”只删除前端列表，没有真正后端删除接口

### 模型配置

1. `SettingsView` 管理多个 OpenAI 兼容服务商
2. 配置保存在浏览器 `localStorage`
3. 配置包含 4 个阶段绑定：
   - collection
   - audit
   - sandbox
   - worldline
4. 所有 API 请求会自动把这份配置编码进请求头 `X-Wolong-Config`
5. 后端依赖 `get_wolong_config` 解析该请求头

## 6. 数据模型关系

核心表及作用如下：

- `cases`：案例主表
- `search_strategies`：关键词策略
- `collection_tasks`：采集任务与进度
- `evidences`：原始证据片段
- `facts`：提炼后的事实卡片
- `fact_audit_alerts`：事实的偏差/风险提示
- `fact_feedbacks`：用户对事实的反馈
- `sandbox_graphs`：因果图快照
- `sandbox_interventions`：沙盘干预历史
- `user_contexts`：用户个人约束
- `worldlines`：世界线结构
- `worldline_interventions`：世界线干预历史
- `whitepapers`：白皮书结果
- `case_feedbacks`：用户对整个案例或白皮书的反馈

整体数据流是：

`Case -> SearchStrategy -> CollectionTask -> Evidence -> Fact -> SandboxGraph -> UserContext -> Worldline -> Whitepaper`

## 7. 当前代码中的关键实现特征

### 7.1 强兜底机制

当前项目为了保证界面“始终可展示”，大量使用默认种子数据兜底：

- 创建案例时立即写入默认事实、证据、图谱、世界线和白皮书
- 当 LLM 生成失败时，再次回退到默认数据

结果是：系统演示稳定性高，但“真实案例内容”和“默认演示内容”容易混在一起。

### 7.2 配置通过请求头下发

前端没有把模型服务商配置持久化到后端，而是：

1. 浏览器保存配置
2. 每次请求自动把配置塞进 Header
3. 后端按 Header 决定本次调用哪个模型

这种方式部署简单，但服务端是无状态的，调试时需要特别注意请求来源。

### 7.3 采集是外部子进程，不是内嵌 SDK

后端不是直接 import 某个平台爬虫类，而是：

1. 拼接命令行参数
2. 启动 MediaCrawler 子进程
3. 等 JSONL 文件生成
4. 再回读 JSONL

这意味着采集链路是“文件中转型”架构。

## 8. 当前实现与设计目标的主要差异

下面这些差异非常关键，影响对项目现状的判断。

### 8.1 案例创建后会直接拥有白皮书

因为 `create_case` 里调用了 `ensure_case_seed_data`，所以新案例刚创建就已经有白皮书、世界线和沙盘数据。

连带影响：

- `cases/history` 里的状态判断会优先命中 `WHITE_PAPER`
- 新案例看起来像是“已经完成推演”

### 8.2 采集完成后的自动跳转存在路由名错误

采集页完成后会跳转到 `InfoPoolSummary`，但路由表中没有这个名字。

结果：

- 自动跳转可能失败
- 用户需要手动点击“边等边看”进入 `FactsPool`

### 8.3 历史恢复里 `COLLECTING` 路由也写错了

历史页恢复 `COLLECTING` 状态时跳转的是 `/workflow/progress`，而真实路径是 `/workflow/collection`。

### 8.4 部分 LLM 输出协议与代码期望不一致

当前提示词与后端解析结构存在明显错位：

- 事实提炼 prompt 要求返回 `knowledge_cards`
- 代码却期望 `list[dict]`
- 沙盘 prompt 要求 `direct_causal_edges`
- 代码却期望 `nodes/edges`
- 世界线 prompt 要求 `worldlines`
- 代码却期望 `timeline/paths`
- 白皮书 prompt 要求 `principal_contradiction`
- 代码却期望 `main_conflict`

结果：

- 即使模型调用成功，也可能因为字段不匹配而被判定失败

### 8.5 若直接调用路由函数，测试当前不通过

我实际执行了 `backend` 下的 `pytest -q`，结果失败。

失败原因：

- 测试直接调用 `routes.generate_search_strategy(case_id, db)`
- 但该函数依赖 `config: WolongConfig = Depends(get_wolong_config)`
- 直接函数调用拿到的是 `Depends` 对象，不是真实配置

这说明当前测试与 FastAPI 依赖注入方式不匹配。

## 9. 结论

当前仓库已经具备一个完整的“决策工作流原型”外壳：

- 前端工作流完整
- 后端数据表完整
- 爬虫接入链路存在
- LLM 调用框架存在
- 沙盘、世界线、白皮书页面可跑通

但从“真实业务闭环”角度看，当前更接近：

1. 可演示的端到端原型
2. 局部已接真实采集和真实大模型
3. 多处仍依赖默认种子数据兜底
4. 提示词协议、路由细节和测试适配仍需校正

如果后续要进入可用版本，优先级最高的工作是：

1. 统一 Prompt 输出协议与后端 DTO
2. 去掉创建案例时对白皮书/世界线的预置写入
3. 修正前端工作流中的错误路由
4. 让采集后的事实提炼真正基于真实证据生成
5. 把测试从“直接调函数”改成 FastAPI TestClient 级别
