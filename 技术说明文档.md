# 卧龙决策支持系统 技术说明书

## 1. 技术目标与架构原则

### 1.1 技术目标

系统需要同时满足以下核心目标，并针对个人开发者和本地部署进行优化：

* **Grounded**：输出必须可追溯到证据片段（Span 级别）。
* **Explainable**：结论、因果关系、行动建议必须可解释，坚决杜绝黑盒逻辑。
* **Controllable**：模型提示词、降级阈值、信息约束均可配置。
* **Agile & Local**：极简技术栈，All in Python，剔除重型中间件，最大化利用 AI 辅助生成代码。

### 1.2 架构指导思想

**前后端分离（Vue 3 + FastAPI） + 模块化单体后端 + 进程内/单文件存储**
当前系统的核心壁垒在于 **LLM 工作流编排、认知审计逻辑和因果图谱的降维处理**。采用最熟悉的 Python 生态和轻量级组件，不仅能快速验证产品核心闭环，也能在依赖 AI 辅助开发时，获得极高的代码生成准确率。

## 2. 总体架构

```text
Frontend Web
  └─ Vue 3 + TypeScript App
      ├─ 决策案例配置（含搜商生成台）
      ├─ 信息池与审计台（事实溯源）
      ├─ 因果沙盘（交互与推演）
      ├─ 世界线演化（个人语境注入）
      └─ 破局白皮书

API Layer
  └─ FastAPI Gateway
      ├─ Case API (案例管理)
      ├─ Agent API (搜商与策略生成)
      ├─ Graph API (NetworkX 图算封装)
      └─ Simulation API (干预与世界线)

Workflow Layer (Orchestrator)
  ├─ Search Strategy Agent (大模型搜商)
  ├─ Source Adapter (爬虫与清洗)
  ├─ L1 Fusion Engine (聚类融合)
  ├─ L2 Audit Engine (认知审计)
  ├─ Causal Sandbox Engine (图计算与解释)
  ├─ Context Injector (个人语境介入)
  ├─ Scenario Engine (演化推演)
  └─ Whitepaper Compiler (报告编译)

Storage Layer (极简本地化)
  ├─ SQLite (通过 SQLAlchemy 管理业务数据与状态)
  ├─ Local File System (原始页面快照/图片)
  ├─ ChromaDB / Qdrant Local (轻量级本地向量检索)
  └─ NetworkX (内存中处理 DAG 拓扑，落库为 JSON)

```

## 3. 技术栈选型

| 层级 | 敏捷版选型 | 核心优势说明 |
| --- | --- | --- |
| **前端框架** | Vue 3 (Composition API) + TS | 语法直观，组件化清晰，AI 工具生成前端 UI 的成功率极高。 |
| **图谱渲染** | ECharts 或 G6 | 生态成熟，支持将后端的 JSON 结构直接渲染为因果图和分支树。 |
| **后端框架** | Python 3.11 + FastAPI | 原生支持异步，契合 LLM 流式输出（SSE），Pydantic 类型校验极强。 |
| **主关系库** | SQLite + SQLAlchemy 2.x | 单文件部署，零运维成本，足以支撑个人级的数据吞吐。 |
| **向量检索** | ChromaDB (本地模式) | Python 原生库，数据存在本地文件夹，无需 Docker 容器。 |
| **图数据库** | NetworkX (Python 库) | 替代沉重的 Neo4j，在内存中完成图计算和拓扑排序，存为 JSON。 |
| **异步队列** | FastAPI BackgroundTasks | 直接复用框架自带功能处理耗时任务（如爬虫、大模型批量提炼）。 |

## 4. 核心模块与大模型引擎设计

系统以 LLM 为核心引擎，划分为不同的能力槽位：

### 4.1 Search Strategy Agent (搜商引擎)

* **职责**：接收初始决策情景，由大模型生成带限制性定语的搜索策略。
* **约束**：必须输出结构化 JSON 列表，防止泛化搜索。支持前端用户微调。

### 4.2 L1 Fusion & L2 Audit (融合与审计双擎)

* **L1 融合**：对相似来源提取 `FactAtom`（原子事实），必须绑定 `evidence_span_ids`。保留条件差异（如地域、时间窗口等边界条件）。
* **L2 审计**：对提取的事实进行“找茬”，标记潜在的幸存者偏差、情绪发泄、隐藏前提，并尽力在库中寻找反证。

### 4.3 Causal Sandbox Engine (因果沙盘引擎)

* **图结构化**：大模型负责将复杂事实转化为 `(节点A) -[正向/负向]-> (节点B)` 的逻辑对。
* **拓扑计算**：后端利用 `NetworkX` 组装有向无环图（DAG），当用户在前端改变某节点状态时，由 NetworkX 执行状态向下游的定性传播。
* **可解释反馈**：计算完成后，将受影响的局部图结构喂给大模型，由其翻译成人类易懂的“逻辑解释与风险警告”。

### 4.4 Scenario Engine & Context Injector (世界线引擎与个人语境)

* **语境隔离**：用户输入的“自身情况”（例如特定目标城市、技能现状）作为“先验节点”单独管理，不污染全局客观事实库。
* **分支演化**：结合因果沙盘和个人语境，推演 Baseline（基准线）、Best-case（理想线）和 Worst-case（恶化线）。

## 5. 数据模型设计 (适配 SQLite 扁平化)

为了摆脱图数据库，我们将复杂的图结构直接序列化存储：

### 5.1 核心实体表

| 实体表 | 核心字段示例 | 说明 |
| --- | --- | --- |
| `Case` | `id`, `profile`, `status` | 记录案例基础状态 |
| `SearchPlan` | `case_id`, `keywords` (JSON) | 存放生成的搜索词和执行进度 |
| `SourceDoc` | `id`, `url`, `content_path` | 原文信息，内容存在本地文件系统 |
| `FactAtom` | `id`, `content`, `evidence_ids` (JSON) | 核心原子事实，必须包含溯源 ID |
| `UserContext` | `case_id`, `constraints` (JSON) | 用户补充的个人现状与硬约束 |
| `CausalGraph` | `case_id`, `graph_data` (JSON) | 整张因果图的快照数据 |

### 5.2 JSON 序列化图谱示例

使用 JSON 存储因果图和用户语境，方便前端直接渲染和模型理解：

```json
{
  "graph_id": "g_001",
  "nodes": [
    {"id": "n1", "label": "全职备考时间", "status": "high", "type": "objective"},
    {"id": "n2", "label": "应届生身份优势", "status": "decaying", "type": "objective"},
    {"id": "c1", "label": "目标地域:潍坊", "status": "fixed", "type": "user_context"}

  "edges": [

}

```

## 6. 工作流与状态机

核心工作流按照你设计的端到端旅程展开：

1. **CREATED**: 用户创建情景。
2. **STRATEGY_GENERATING**: `Search Agent` 生成搜索词，用户确认。
3. **COLLECTING**: 执行合规爬取与本地存储。
4. **FUSING_AND_AUDITING**: 执行 L1 去重提炼与 L2 认知审计。
5. **EVIDENCE_GATE**: 检查样本量，若冲突过大或证据不足，状态转为 `INSUFFICIENT` 并提示用户补充。
6. **GRAPH_BUILDING**: 生成 `FactAtom` 对应的 DAG 并存入 `CausalGraph` 表。
7. **SIMULATING**: 接收用户干预，通过 NetworkX 重算，输出反事实推演。
8. **CONTEXT_INJECTING**: 收集用户的实际约束与补充信息。
9. **WORLDLINE_GENERATING**: 生成不同时间节点下的系统演化线。
10. **COMPLETED**: 编译最终的《破局白皮书》。

## 7. 核心 API 接口设计 (FastAPI)

所有长耗时接口推荐采用前端轮询任务状态或 SSE (Server-Sent Events) 流式返回。

| HTTP 方法 | 路径 | 核心作用 |
| --- | --- | --- |
| `POST` | `/api/v1/cases` | 初始化决策案例 |
| `POST` | `/api/v1/cases/{id}/search-strategy` | 触发搜商大模型生成关键词列表 |
| `GET` | `/api/v1/cases/{id}/facts` | 获取带有溯源标记的事实卡片列表 |
| `GET` | `/api/v1/cases/{id}/sandbox` | 获取当前案例的因果图 JSON 结构 |
| `POST` | `/api/v1/cases/{id}/sandbox/intervene` | 提交变量状态修改，返回图谱重算结果 |
| `PUT` | `/api/v1/cases/{id}/context` | 更新用户的个人情况与硬约束信息 |
| `GET` | `/api/v1/cases/{id}/whitepaper` | 获取最终生成的行动建议和白皮书 |

## 8. 降级与安全风控限制

* **模型幻觉控制**：所有生成类的 Prompt 必须带有强指令：“如果提供的信息池中没有相关依据，请回复‘缺乏足够证据’，绝不允许根据预训练知识进行编造。”
* **溯源强制门槛**：白皮书中提取的每一条事实或建议，在后端组装时如果找不到对应的 `evidence_ids`，编译器将直接予以抛弃。
* **定性优于定量**：禁止系统输出“成功率为 35%”等伪精确概率。全部采用高/中/低置信度，并在沙盘中明确标识逻辑冲突的点。

## 9. 敏捷演进建议 (执行路线图)

考虑到全栈开发的精力分配，建议利用 AI 编程工具，按照以下步骤逐步攻克：

* **Phase 1：信息管道与审计台 (攻克数据流)**
* 搭建 FastAPI + SQLite 环境，跑通前后端通信。
* 利用大模型完成搜商策略生成 -> 模拟数据接入 -> 完成 JSON 格式的原子事实提炼和认知谬误标记。
* 前端重点：事实卡片展示与证据高亮溯源。


* **Phase 2：因果沙盘与图计算 (攻克逻辑流)**
* 引入 `NetworkX` 库，在后端把事实转化为有向无环图。
* 打通前端拖拽/修改节点变量 -> 后端图计算 -> 模型输出推演结果的交互闭环。


* **Phase 3：世界线与个人语境 (攻克决策闭环)**
* 增加用户独立上下文表单。
* 基于前两个阶段的成果，利用模型聚合输出结构化的《破局白皮书》和 MVP 行动建议。
